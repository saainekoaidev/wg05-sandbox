# ユーザー登録画面 設計書

## 1. 画面概要

| 項目 | 内容 |
| --- | --- |
| 画面ID | （未定） |
| 画面名 | ユーザー登録画面 |
| 概要 | 新規ユーザーがアカウント情報および基本情報を登録するための画面 |
| URL | `/register`（仮） |
| アクセス権限 | 未認証ユーザー |

---

## 2. 画面レイアウト

```
┌─────────────────────────────────────────┐
│                                         │
│  メルアド  [____________________]       │
│                                         │
│  パスワード [____________________]       │
│                                         │
│  ─────────────────────────────────       │
│                                         │
│  なまえ   [____________________]        │
│                                         │
│  〒       [____________________]        │
│                                         │
│                                         │
│   [ 登録 ]   [ リセット ]   [ キャンセル ] │
│                                         │
└─────────────────────────────────────────┘
```

画面は上下2ブロックに分かれる。上ブロックがアカウント情報（メールアドレス・パスワード）、下ブロックがユーザー基本情報（氏名・郵便番号）。区切り線で視覚的に分離する。

---

## 3. 入力項目定義

| No | 項目名 | 表示ラベル | 入力形式 | 必須 | 桁数 | バリデーション | 備考 |
| --- | --- | --- | --- | :-: | --- | --- | --- |
| 1 | メールアドレス | メルアド | テキスト | ◯ | 最大256文字 | RFC5322 形式チェック / 重複チェック | ログインIDとして使用 |
| 2 | パスワード | パスワード | パスワード（マスク表示） | ◯ | 8〜32文字 | 半角英数記号、英大文字・小文字・数字を各1文字以上含む | 入力値はマスク表示 |
| 3 | 氏名 | なまえ | テキスト | ◯ | 最大50文字 | 文字種チェック（全角可） | 姓名一体で入力 |
| 4 | 郵便番号 | 〒 | テキスト | ◯ | 7桁（ハイフン任意） | 半角数字、日本の郵便番号形式 | 例: `1000001` または `100-0001` |

---

## 4. ボタン定義

| No | ボタン名 | 表示ラベル | 動作 |
| --- | --- | --- | --- |
| 1 | 登録ボタン | とうろく | 入力内容のバリデーションを実行し、問題がなければサーバへ送信。登録完了後、完了画面へ遷移する |
| 2 | リセットボタン | リセット | 入力中の全項目をクリアし、初期状態に戻す（確認ダイアログを表示） |
| 3 | キャンセルボタン | キャンセル | 登録処理を中断し、前画面（またはトップ画面）へ遷移する |

> ※スケッチ上3つ目のボタン名が判読困難なため、「キャンセル」を仮置き。「戻る」「ログイン画面へ」など要件に応じて修正のこと。

---

## 5. 画面遷移

| イベント | 遷移先 | 条件 |
| --- | --- | --- |
| 登録ボタン押下（成功時） | 登録完了画面 | 全バリデーションOK かつ サーバ登録成功 |
| 登録ボタン押下（失敗時） | 本画面（エラー表示） | バリデーションNG または サーバエラー |
| リセットボタン押下 | 本画面（初期化） | 確認ダイアログで「OK」選択時 |
| キャンセルボタン押下 | 前画面 | 入力内容がある場合は確認ダイアログを表示 |

---

## 6. バリデーション仕様

### 6.1 クライアントサイド

入力フォーカスアウト時および登録ボタン押下時に実行する。エラー時は対象項目の直下に赤字でエラーメッセージを表示する。

### 6.2 サーバサイド

クライアントサイドと同等のチェックに加え、メールアドレスの重複チェックを行う。

### 6.3 エラーメッセージ

| 項目 | 条件 | メッセージ |
| --- | --- | --- |
| メールアドレス | 未入力 | メールアドレスを入力してください |
| メールアドレス | 形式不正 | メールアドレスの形式が正しくありません |
| メールアドレス | 重複 | このメールアドレスは既に登録されています |
| パスワード | 未入力 | パスワードを入力してください |
| パスワード | 桁数不足/超過 | パスワードは8〜32文字で入力してください |
| パスワード | 文字種不足 | パスワードは英大文字・小文字・数字をそれぞれ1文字以上含めてください |
| 氏名 | 未入力 | 氏名を入力してください |
| 郵便番号 | 未入力 | 郵便番号を入力してください |
| 郵便番号 | 形式不正 | 郵便番号は7桁の数字で入力してください |

---

## 7. その他考慮事項

- パスワード入力欄には表示/非表示を切り替えるアイコンを設置する
- 郵便番号入力後、住所自動補完APIと連携する場合は別途検討（現状の画面イメージには住所欄なし）
- 登録処理中はボタンを非活性化し、二重送信を防止する
- スマートフォン表示時は項目を縦1列に並べ、ボタンも横幅いっぱいの縦並びとする

---

## 8. 未確定事項

スケッチから判読できない、もしくは仕様確認が必要な項目を以下に列挙する。

- 3つ目のボタン名（「キャンセル」「戻る」「ログイン画面へ」のいずれか）
- 住所欄の有無（〒のみで住所欄がないため、住所が必要かどうか）
- 入力ラベルの正式表記（「メルアド」「なまえ」などスケッチのままか、「メールアドレス」「氏名」など正式名称にするか）
- 認証方式（メール認証の有無、認証メール送信フローの要否）
