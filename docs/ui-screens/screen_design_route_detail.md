## 4. ボタン定義

| No | ボタン名 | 表示ラベル | 動作 |
| --- | --- | --- | --- |
| 1 | 編集ボタン | 編集 | 経路編集画面 ([screen_design_route_edit.md](screen_design_route_edit.md)) `/routes/:id/edit` へ遷移 |
| 2 | 削除ボタン | 削除 | 確認ダイアログ表示 → `DELETE /api/routes/:id` を呼出。成功なら経路一覧画面 ([screen_design_route_list.md](screen_design_route_list.md)) へ遷移 |
| 3 | 一覧に戻るボタン | 一覧に戻る | 経路一覧画面 (`/routes`) へ遷移 |

> ボタンは `.btn` クラスを使用し、`<a>` でリンク化する場合も `text-align: center` / `text-decoration: none` / `inline-flex` 中央寄せが効くように共通CSSで吸収済み。

## 5. 画面遷移

| イベント | 遷移先 | 条件 |
| --- | --- | --- |
| 編集ボタン押下 | 経路編集画面 (`/routes/:id/edit`) | 必ず |
| 削除ボタン押下（成功） | 経路一覧画面 (`/routes`) | 確認ダイアログ「OK」かつ DELETE 成功 |
| 削除ボタン押下（失敗） | 本画面（エラー表示） | 確認ダイアログ「OK」かつ DELETE 失敗 |
| 一覧に戻るボタン押下 | 経路一覧画面 (`/routes`) | 必ず |
| 不正なID / Not Found | 経路一覧画面（エラー表示） | GET 404 / 403 |
| 未認証アクセス | ログイン画面 (`/login`) | セッション無効 |

## 6. バリデーション仕様

### 6.1 クライアントサイド

削除ボタン押下時に「この経路を削除しますか?」確認ダイアログを表示し、「OK」選択時のみ `DELETE` を発行する。本画面に入力項目はないためフォームバリデーションは存在しない。

### 6.2 サーバサイド

`GET /api/routes/:id` および `DELETE /api/routes/:id` は認証必須。Route の `userId` がセッションユーザーと一致することを検証する（オーナーチェック）。

### 6.3 エラーメッセージ

| 項目 | 条件 | メッセージ |
| --- | --- | --- |
| 取得 | 通信失敗 | 経路の取得に失敗しました。再読み込みをお試しください |
| 取得 | 対象なし (404) | 該当の経路が見つかりませんでした |
| 取得 | 権限なし (403) | この経路を表示する権限がありません |
| 削除 | 通信失敗 | 経路の削除に失敗しました。再度お試しください |
| 削除 | 権限なし (403) | この経路を削除する権限がありません |
| 削除 | 対象なし (404) | 該当の経路が見つかりませんでした（既に削除されている可能性があります） |

## 7. その他考慮事項

- 編集 / 削除ボタンはオーナーのみに表示する想定（将来共有機能追加時の分岐ポイント）
- 印刷時用のレイアウト（CSS `@media print`）は次フェーズで検討
- 区間明細は `orderIndex` 昇順で表示
- 種別タグは色分けで視認性を高める（`tag-train` 青 / `tag-subway` 橙 / `tag-bus` 緑 / `tag-other` 灰）
- 路線名 (`lineName`) が `null` または空文字の場合は、その区間の路線名行を表示しない（タグだけ表示）
- 共有用URL発行（read-only リンク）は本画面のスコープ外
- 削除実行後の戻り先は経路一覧。経路詳細から削除した直後に「削除しました」のバナーを一覧に引き継ぐ（state 経由）

## 8. 未確定事項

- 削除確認はダイアログのみで良いか（パスワード再入力等の追加チェック要否）
- 過去の更新履歴（diff）表示の要否
- 共有機能（読み取り専用URLの発行）の要否
- 月間出勤日数を入力して総額を算出する簡易計算機能の要否
- 種別ごとの追加メタ情報（バスの系統番号、便名など）の表示有無
- 路線名が複数の路線にまたがる場合の表示方法（現状は単一文字列）
