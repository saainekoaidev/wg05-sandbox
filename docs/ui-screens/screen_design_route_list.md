## 3. 表示項目定義

| No | 項目名 | 表示ラベル | 種別 | 桁数 | 備考 |
| --- | --- | --- | --- | --- | --- |
| 1 | No | # | 整数（行番号） | - | 1始まりの連番（DB主キーではなく表示用） |
| 2 | 経路名 | 経路名 | テキスト | 最大50文字 | 任意項目。未設定時は「(無題)」を表示 |
| 3 | 種別 / 路線 | 種別 / 路線 | タグ + テキスト | - | 区間中の `kind` ユニーク集合を色分けタグで上段に、`lineName` ユニーク集合を `/` 区切りで下段に表示（cell-stack） |
| 4 | 出発→到着 | 出発 → 到着 | テキスト | 各最大50文字 | `渋谷 → 神田` の形式で連結表示 |
| 5 | 区間数 | 区間 | 整数 | 1〜10 | Segment 件数 |
| 6 | 合計運賃 | 合計運賃 | 整数（円） | 0〜999,999 | 各区間運賃の合計、`¥` プレフィックス + 3桁区切り |
| 7 | 操作 | 操作 | ボタン群 | - | 詳細 / 編集 / 削除 |

### 種別 / 路線セル の生成ルール

- **種別タグ**: `route.segments[].kind` の値をユニーク化し、`['train', 'subway', 'bus', 'other']` の順で並べ、各値に対応するタグクラス (`tag-train` / `tag-subway` / `tag-bus` / `tag-other`) で表示
- **路線名リスト**: `route.segments[].lineName` のうち空でないものをユニーク化し、出現順に `/` で区切って表示。全区間で `lineName` が空の場合は路線名行を非表示
- いずれもクライアント側で算出（API レスポンスの `segments[]` から派生）

## 5. 画面遷移

| イベント | 遷移先 | 条件 |
| --- | --- | --- |
| 新規登録ボタン押下 | 経路登録画面 (`/routes/new`) | 必ず |
| 詳細ボタン押下 | 経路詳細画面 (`/routes/:id`) | 必ず（押下行の id を渡す） |
| 編集ボタン押下 | 経路編集画面 (`/routes/:id/edit`) | 必ず |
| 削除ボタン押下（成功） | 本画面（再描画） | 確認ダイアログで「OK」かつ DELETE 成功 |
| 削除ボタン押下（失敗） | 本画面（エラー表示） | 確認ダイアログで「OK」かつ DELETE 失敗 |
| ログアウトボタン押下 | ログイン画面 (`/login`) | 必ず |
| 未認証アクセス | ログイン画面 (`/login`) | セッション無効 |

## 6. バリデーション仕様

### 6.1 クライアントサイド

削除ボタン押下時に「この経路を削除しますか?」確認ダイアログを表示し、「OK」選択時のみ `DELETE` を発行する。

### 6.2 サーバサイド

`GET /api/routes` および `DELETE /api/routes/:id` は認証必須。さらに DELETE では対象 Route の `userId` がセッションユーザーと一致することを検証する（オーナーチェック）。

### 6.3 エラーメッセージ

| 項目 | 条件 | メッセージ |
| --- | --- | --- |
| 一覧取得 | 通信失敗 | 経路一覧の取得に失敗しました。再読み込みをお試しください |
| 削除 | 通信失敗 | 経路の削除に失敗しました。再度お試しください |
| 削除 | 権限なし (403) | この経路を削除する権限がありません |
| 削除 | 対象なし (404) | 該当の経路が見つかりませんでした（既に削除されている可能性があります） |
| 削除 | 成功 | 経路を削除しました（成功メッセージは画面上部にバナー表示） |

## 7. その他考慮事項

- 経路ゼロ件時は表本体の代わりに `.empty` で空状態メッセージ「まだ通勤経路が登録されていません」と「+ 新規登録」誘導を表示する
- 並び順はデフォルトで `createdAt` 降順（新しいものが上）
- 削除はソフトデリートではなく物理削除を前提とする（教育目的の単純化）
- 行数が増えた場合のページング・無限スクロールは初期実装には含めない（10件未満想定）
- 合計運賃は API レスポンスに含めず、フロント側で `segments[].fare` を合計して算出する（将来 Prisma の集約に置換検討）
- 種別タグおよび路線名リストもクライアント側で算出（API は `segments[]` をそのまま返す）
- 種別タグの並び順は `train` → `subway` → `bus` → `other` で固定し、利用者にとって視覚的に同じ意味になるようにする
