- ヘッダタイトル: 「通勤経路の編集」

- 主要アクションが「登録する」→「更新する」

## 3. 入力項目定義

経路登録画面 ([screen_design_route_register.md](screen_design_route_register.md)) の「3. 入力項目定義」と同等（経路名 / 区間種別 / 区間路線名 [select] / 区間出発駅 / 区間到着駅 / 区間運賃）。経路全体の出発駅・到着駅は同様に **派生項目**（`segments[0].fromStation` / `segments[last].toStation` から自動算出）。

初期値は `GET /api/routes/:id` のレスポンス（`name`, `segments[].{kind, lineName, fromStation, toStation, fare}`）をフォームへプリフィルする。`Route.fromStation` / `Route.toStation` は API から取得しても表示用のサマリ算出にはクライアントが segments から再算出した値を用いる（不整合があった場合のフェイルセーフ）。

差分検知のため、初期値のスナップショットを保持し、未変更時は更新ボタンを非活性化する。差分判定は経路名 / 各区間の全フィールド（`kind` / `lineName` / `fromStation` / `toStation` / `fare`）+ 区間の追加・削除・順序を対象とする。

## 4. ボタン定義

| No | ボタン名 | 表示ラベル | 動作 |
| --- | --- | --- | --- |
| 1 | 更新ボタン | 更新する | 全項目バリデーションを実行し、問題なければ `PATCH /api/routes/:id` を呼出。成功なら経路詳細画面 (`/routes/:id`) へ遷移 |
| 2 | キャンセルボタン | キャンセル | 経路詳細画面 (`/routes/:id`) へ戻る。変更がある場合は確認ダイアログ |
| 3 | 駅選択ボタン | 駅選択 | 駅マスタ参照画面 ([screen_design_station_master.md](screen_design_station_master.md)) を **ポップアップウィンドウ** として開く。仕様は経路登録画面 ([screen_design_route_register.md](screen_design_route_register.md)) と同一 |
| 4 | 区間追加ボタン | + 区間を追加 | 区間行を1行追加（最大10行まで）。追加後はサマリバーを再計算 |
| 5 | 区間削除ボタン | × | 該当区間行を削除（最低1行は残す）。削除後はサマリバーを再計算 |

> 削除アクションは経路詳細画面 ([screen_design_route_detail.md](screen_design_route_detail.md)) と経路一覧画面 ([screen_design_route_list.md](screen_design_route_list.md)) に集約し、本画面からは行わない（誤操作防止）。

## 5. 画面遷移

| イベント | 遷移先 | 条件 |
| --- | --- | --- |
| 更新ボタン押下（成功） | 経路詳細画面 (`/routes/:id`) | バリデーションOK + 更新成功 |
| 更新ボタン押下（失敗） | 本画面（エラー表示） | バリデーションNG / 楽観ロック衝突 / サーバエラー |
| キャンセルボタン押下 | 経路詳細画面 (`/routes/:id`) | 変更なし、または確認ダイアログ「OK」 |
| 駅選択ボタン押下 | 駅マスタ参照ポップアップ | 必ず |
| 駅選択完了 | 本画面（該当入力欄に駅名反映 + サマリ再計算） | `postMessage` 受信時 |
| 不正なID / Not Found | 経路一覧画面（エラー表示） | GET 404 / 403 |
| 未認証アクセス | ログイン画面 (`/login`) | セッション無効 |

## 6. バリデーション仕様

### 6.1 クライアントサイド

経路登録画面と同等。加えて、未変更時は更新ボタンを非活性化（差分検知）。

### 6.2 サーバサイド

経路登録画面と同等のチェックに加え、以下を検証する。
- 認証 / オーナーチェック（`Route.userId === session.userId`）
- 楽観ロック（リクエストに `updatedAt` を含め、サーバ側の最新値と一致しない場合 409）
- 受信した segments から `Route.fromStation` / `Route.toStation` を再計算して保存

### 6.3 エラーメッセージ

経路登録画面 ([screen_design_route_register.md](screen_design_route_register.md)) の「6.3 エラーメッセージ」と同等（種別未選択 / 路線名不正 / 区間運賃 / 連結性 / 派生駅同一 等を含む）。加えて以下を追加する。

| 項目 | 条件 | メッセージ |
| --- | --- | --- |
| 楽観ロック | 衝突 (409) | 他の場所で更新されたため最新の状態を再読込しました。再度ご確認ください |
| 取得 | 対象なし (404) | 該当の経路が見つかりませんでした |
| 取得 | 権限なし (403) | この経路を編集する権限がありません |
| 通信 | 更新失敗 | 経路の更新に失敗しました。時間をおいて再度お試しください |

## 7. その他考慮事項

- フォーム送信時は `If-Match` ヘッダもしくはリクエストボディで `updatedAt` を送信し、楽観ロックを実装する
- 変更が無い場合は更新ボタン非活性化（差分検知。区間の `kind` / `lineName` の変更、区間の追加・削除・順序変更も差分扱い）
- 経路の出発駅・到着駅は **入力項目ではなく派生項目**。プリフィル時もサマリバーは segments から再計算
- ID は URL パラメータから取得し、隠しフィールドとしては持たない
- 楽観ロック衝突時は最新値を再取得してフォームを再描画し、ユーザーに変更内容の再確認を促す
- 駅選択ボタンの動作（ポップアップ + `postMessage`）は経路登録画面と同一実装を共有
- 編集中の自動保存（draft）は本フェーズでは実装しない
