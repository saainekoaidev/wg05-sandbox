- 区間は `.segment-card`（上段: 番号 / 種別 / 路線名 / 削除ボタン、下段: 出発駅 / 到着駅 / 運賃）で表現する。区間カードは `.segments` ボックス内に縦積み

## 3. 入力項目定義

| No | 項目名 | 表示ラベル | 入力形式 | 必須 | 桁数 | バリデーション | 備考 |
| --- | --- | --- | --- | :-: | --- | --- | --- |
| 1 | 経路名 | 経路名 | テキスト | △ | 最大50文字 | 文字種チェック（全角可） | 任意。未設定時は「(無題)」として保存 |
| 2 | 区間種別 | 種別 | セレクトボックス | ◯ | - | `train` / `subway` / `bus` / `other` のいずれか | 既定値 `train`。各区間に必須 |
| 3 | 区間路線名 | 路線名 | **セレクトボックス** | △ | - | 路線リストの値、または `その他` | 既定値「(未選択)」。未選択は `null` として保存 |
| 4 | 区間出発駅 | 出発（区間） | テキスト + 駅選択 | ◯ | 最大50文字 | 区間ごとに必須 / 文字種チェック（全角可） | 各区間1件 |
| 5 | 区間到着駅 | 到着（区間） | テキスト + 駅選択 | ◯ | 最大50文字 | 同上 / 区間内で出発と同一不可 | 各区間1件 |
| 6 | 区間運賃 | 運賃 | 数値（半角整数） | ◯ | 1〜99,999（円） | 1以上の整数 | 単位は円 |

### 3.1 区間に関する追加規則

- 区間数は最低1、最大10
- 隣接区間の連結性（前区間の到着駅 = 次区間の出発駅）は **警告レベル** とし、登録は許容する
- 種別 (`kind`) は表示色分け用のキーとしても利用する（`tag-train` / `tag-subway` / `tag-bus` / `tag-other`）

### 3.2 自動算出項目（読み取り専用）

| 項目 | 算出ルール | 表示位置 |
| --- | --- | --- |
| 経路の出発駅 | `segments[0].fromStation`（1区間目の出発駅） | サマリバー左 |
| 経路の到着駅 | `segments[N-1].toStation`（最終区間の到着駅） | サマリバー左 |
| 合計運賃 | `Σ segments[].fare`（区間運賃の合計、未入力区間は0として加算） | サマリバー右 |

入力中はリアルタイム再計算（`oninput` / `onchange`）。未入力の場合は `(未入力)` と薄く表示する。

### 3.4 Segment データモデルとの対応

API へ送信する `segments[]` 各要素は以下のキーを持つ。サーバ側の Prisma `Segment` モデルへ追加が必要なフィールドを含む。

| キー | 型 | 必須 | 備考 |
| --- | --- | :-: | --- |
| `kind` | `"train" \| "subway" \| "bus" \| "other"` | ◯ | 種別（enum / リテラル文字列） |
| `lineName` | `string?` | △ | 路線名（最大50文字、未選択時 `null`） |
| `fromStation` | `string` | ◯ | 区間出発駅（最大50文字） |
| `toStation` | `string` | ◯ | 区間到着駅（最大50文字） |
| `fare` | `number` | ◯ | 区間運賃（1〜99,999） |
| `orderIndex` | `number` | ◯ | サーバ側で配列順から自動採番（クライアントは送信不要） |

`Route.fromStation` / `Route.toStation` はクライアントが送信せず、サーバ側で `segments[0].fromStation` / `segments[last].toStation` から派生して保存する（または送信しても再計算で上書きする）。

## 4. ボタン定義

| No | ボタン名 | 表示ラベル | 動作 |
| --- | --- | --- | --- |
| 1 | 登録ボタン | 登録する | 全項目バリデーションを実行し、問題なければ `POST /api/routes` を呼出。成功なら経路詳細画面 ([screen_design_route_detail.md](screen_design_route_detail.md)) へ遷移 |
| 2 | リセットボタン | リセット | 入力中の全項目をクリア（区間は1行に戻す）。確認ダイアログを表示。サマリバーも再計算 |
| 3 | キャンセルボタン | キャンセル | 経路一覧画面 ([screen_design_route_list.md](screen_design_route_list.md)) へ遷移。入力ありの場合は確認ダイアログ |
| 4 | 駅選択ボタン | 駅選択 | 駅マスタ参照画面 ([screen_design_station_master.md](screen_design_station_master.md)) を **ポップアップウィンドウ** として開く。対象入力欄を `data-target` 属性で記憶し、ポップアップ内で「選択」されたら `postMessage` 経由で駅名を受け取って当該入力欄に反映する。反映後はサマリバーを再計算 |
| 5 | 区間追加ボタン | + 区間を追加 | 区間行を1行追加（最大10行まで）。新規追加時の既定値は `kind="train"`、`lineName=""(未選択)`、駅・運賃は空。追加後はサマリバーを再計算 |
| 6 | 区間削除ボタン | × | 該当区間行を削除（最低1行は残す）。削除後はサマリバーを再計算 |

### 4.1 駅選択ボタン押下時のポップアップ仕様

```js
// 親画面 (route_register / route_edit) 側の実装イメージ
function openStationPicker(targetInputId) {
  const win = window.open(
    '/stations',
    'station-picker',
    'width=960,height=720,resizable=yes,scrollbars=yes'
  );
  pendingTarget = targetInputId;
}

window.addEventListener('message', (e) => {
  if (e.data?.type === 'station-pick' && pendingTarget) {
    document.getElementById(pendingTarget).value = e.data.name;
    pendingTarget = null;
    recalc();    // サマリバーを更新
  }
});
```

ポップアップがブラウザにブロックされた場合のフォールバックとして、`/stations` への単独画面遷移を許容する（戻り先は親画面 + URLハッシュで復元）。

## 5. 画面遷移

| イベント | 遷移先 | 条件 |
| --- | --- | --- |
| 登録ボタン押下（成功） | 経路詳細画面 (`/routes/:id`) | バリデーションOK + サーバ登録成功 |
| 登録ボタン押下（失敗） | 本画面（エラー表示） | バリデーションNG または サーバエラー |
| リセットボタン押下 | 本画面（初期化） | 確認ダイアログで「OK」選択時 |
| キャンセルボタン押下 | 経路一覧画面 (`/routes`) | 入力なし、または確認ダイアログで「OK」選択時 |
| 駅選択ボタン押下 | 駅マスタ参照ポップアップ | 必ず（ポップアップ内で「選択」または「閉じる」） |
| 駅選択完了 | 本画面（該当入力欄に駅名反映 + サマリ再計算） | `postMessage` 受信時 |
| 未認証アクセス | ログイン画面 (`/login`) | セッション無効 |

## 6. バリデーション仕様

### 6.1 クライアントサイド

入力フォーカスアウト時および登録ボタン押下時に実行する。
- 各フィールド単位のエラー: 入力直下に赤字
- フォーム全体の致命エラー: 画面上部のバナー (`.banner.is-shown`) に表示
- 区間連結性の警告（致命ではない）: 該当区間の下にオレンジ色のヒント表示（仮 `.hint--warn`）

### 6.2 サーバサイド

クライアントサイドと同等のチェックに加え、以下を検証する。
- ユーザー認証状態（未認証は 401）
- 区間運賃の整合性（必須・上限）
- 区間種別の値が enum（`train|subway|bus|other`）に含まれること
- 区間路線名が許容文字列（マスタ値 + `その他` + `null`）であること
- リクエストボディのスキーマ検証（zod）
- 受信した segments から `Route.fromStation` / `Route.toStation` を再計算して保存

同名経路（同一ユーザー内）の重複は許容する。

### 6.3 エラーメッセージ

| 項目 | 条件 | メッセージ |
| --- | --- | --- |
| 経路名 | 桁数超過 | 経路名は50文字以内で入力してください |
| 区間 | 0件 | 区間を1つ以上登録してください |
| 区間 | 上限超過 | 区間は最大10個まで登録できます |
| 区間種別 | 未選択 / 不正な値 | 種別を選択してください |
| 区間路線名 | 不正な値（マスタ外） | 路線名はリストから選択してください |
| 区間出発駅 / 区間到着駅 | 未入力 | 区間ごとに出発駅と到着駅を入力してください |
| 区間出発駅 / 区間到着駅 | 桁数超過 | 駅名は50文字以内で入力してください |
| 区間出発駅・区間到着駅 | 同一 | 区間内で出発駅と到着駅が同じです |
| 区間運賃 | 未入力 | 区間ごとに運賃を入力してください |
| 区間運賃 | 形式不正 | 運賃は1以上の整数で入力してください |
| 区間運賃 | 上限超過 | 運賃は99,999円以下で入力してください |
| 区間連結性 | 隣接区間の駅不一致 | 隣接する区間の駅がつながっていません（このまま登録は可能です） |
| 経路全体（派生） | 1区間目の出発駅 = 最終区間の到着駅 | 経路の出発駅と到着駅が同じです（このまま登録は可能ですが循環経路となります） |
| 通信 | サーバエラー | 経路の登録に失敗しました。時間をおいて再度お試しください |

## 7. その他考慮事項

- 経路の出発駅・到着駅は **入力項目ではなく派生項目**。ユーザーが直接編集することはない
- サマリバーは `oninput` / `onchange` で常時最新の派生値を表示。区間追加・削除・並び替え操作後も再計算
- 路線名はセレクトボックス。マスタ未収載の路線は `その他` を選択 → 将来的にコメント欄や自由入力欄を追加する可能性あり
- 駅選択ボタンはポップアップ (`window.open`) を採用。ブロックされた場合のフォールバックとして単独画面 (`/stations`) への遷移も用意する
- 駅選択結果は `postMessage` で親画面に返却し、対象入力欄を `data-target` 属性で識別。反映後にサマリ再計算
- 合計運賃は区間運賃の変更に応じてリアルタイムに再計算
- 登録処理中はボタンを非活性化し、二重送信を防止する
- 区間の並び替え（ドラッグ操作）は次フェーズ
- 駅マスタに無い駅をフリー入力で登録することを許容する（マスタ未整備時の暫定）
- 種別 (`kind`) は表示時のタグ色分け（`tag-train` / `tag-subway` / `tag-bus` / `tag-other`）に対応する
- スマートフォン表示時は `.segment-card-head` / `.segment-card-route` を縦積みに切替（CSS 既存対応）

## 8. 未確定事項

- 経路名を必須にすべきか
- 種別の選択肢を将来拡張するか（飛行機・船舶・自転車・徒歩 など）
- 種別ごとに必要な追加情報の有無（例: バスの場合の系統番号、飛行機の場合の便名）
- 路線マスタの仕様（DB テーブル化 / 外部API / 種別ごとの絞り込み連動 等）
- 路線リストに無い路線を登録するためのフロー（`その他` 選択 → 自由入力欄を出す等）
- 同一ユーザー内の経路名重複を許容するか
- 区間連結性チェックを警告で良いか、エラーにすべきか
- 出発駅と到着駅が同じ循環経路を許容するか
- 区間並び替え対応の時期
- 月間費用シミュレーター（出勤日数 × 合計運賃 × 2）の要否
- 駅マスタに無い駅の登録を制限すべきか（自由入力可とするか）
- ポップアップブロック時のフォールバック単独画面の戻り先パラメータ仕様
