# 0004 ユーザー情報メンテナンスの対象項目とアカウント削除ポリシー

## Status
Accepted (2026-05-03)

## Context

US-001 〜 US-007 は経路 CRUD のみを対象としており, ユーザー情報の参照・更新・削除を行う機能は実装も要件も無い。better-auth に依存している `User` モデルは現状 `id / name / email / emailVerified / image / createdAt / updatedAt` の 7 項目で, パスワードは `Account.password` に scrypt ハッシュで格納されている (docs/adr/0002-data-model.md §3.1)。

利用者が長期間サービスを使う前提に立つと, 以下のメンテナンスニーズが避けられない:

- 改姓や敬称変更による表示名 (name) の更新
- 引っ越しに伴う属性情報の更新 (※郵便番号は現スキーマには存在しない)
- 定期的な, あるいは情報漏洩懸念時のパスワード変更
- 退会時のアカウント削除と関連データの完全削除

これらを US-008/009/010 として要件に追加するにあたり, **(a) どの項目をメンテ対象とするか**, **(b) 削除時に関連データをどう扱うか** の 2 点を確定させる必要がある。

サンドボックスの目的は工程追体験であり, 通勤経路という業務データの機密度に踏み込んだ高度な要件 (本人確認再認証, ソフト削除と監査ログ, 個人情報マスキング等) は対象外とする。Tier 1 構成 (CLAUDE.md) に収まる範囲で最小限のメンテ機能を提供する。

## Decision

### (a) メンテ対象とする User 項目

`User` モデルに **`postalCode String?`** (任意 7 桁数字, ハイフン無しで保存) を追加する。これは US-008 の「郵便番号変更」要件 (要件定義 US-008) を満たすため。アプリケーション層で `^\d{7}$` 検証を行い, 表示時にハイフン挿入する。

メンテナンス対象は以下に限定する:

| 項目 | 変更可否 | US |
|---|---|---|
| `id` | 不可 (内部 ID) | — |
| `email` | 不可 (このサンドボックスでは scope 外。better-auth の `changeEmail` は本人確認フローを伴うため US 化を見送る) | — |
| `name` | 可 | US-008 |
| `postalCode` (新規) | 可 (任意項目) | US-008 |
| `image` | 不可 (スコープ外) | — |
| `password` (Account.password) | 可 (現パスワード入力必須) | US-009 |
| アカウント自体 | 削除可 (物理削除) | US-010 |

email 変更は技術的に better-auth で可能だが, 本人確認 (古アドレスへの確認メール送信) を必要とし, またメール送信基盤がサンドボックスに無いため, 今回は対象外とする。要件として将来追加する場合は別 US を起票する。

### (b) アカウント削除時のポリシー

物理削除を採用する。論理削除 (deletedAt 列追加) は監査要件が無いサンドボックスではコスト過多なため見送る。

削除時に連鎖的に消えるレコードは以下:

- `Session` (Cascade) — 既存設定通り
- `Account` (Cascade) — 既存設定通り
- `Route` (Cascade) → `RouteSegment` (Cascade)— 既存設定通り
- `Verification` — User と直接の FK は無いが, 該当 identifier (= email) のレコードが残るため削除処理内で明示的に `prisma.verification.deleteMany({ where: { identifier: email } })` を実行する。

マスタ (`Line` / `Station` / `StationLine`) は影響を受けない。

削除 API (`DELETE /api/users/me` 想定) は **現パスワード再入力による再認証** を必須とし, 確認ダイアログでも改めて意思確認を行う。better-auth の `deleteUser` API がパスワード検証を内蔵するためこれを利用する。削除後はセッション cookie も即時失効させ, クライアントは `/login` に遷移する。

### パスワード変更フロー (US-009)

better-auth の `changePassword` (現パスワード + 新パスワード) を利用。新パスワードは US-001 と同じ強度ルール (8 文字以上 + 文字種混在) を適用する。変更成功後は他デバイスのセッションを失効させる (`revokeOtherSessions: true`)。

### 画面構成

`/account` 1 画面に以下のセクションを縦に並べる:

1. プロフィール (name / postalCode の編集 + 「保存」ボタン)
2. パスワード変更 (現パス / 新パス / 確認 + 「変更」ボタン)
3. アカウント削除 (現パス入力 + 「アカウントを削除」ボタン, 確認ダイアログ必須)

ナビゲーションは `/routes` のヘッダ右上 「ユーザー: foo@example.com」 をリンク化して `/account` へ。

## Consequences

利点:
- 利用者がライフイベント (改姓, 引っ越し, 退会) に対応できる, 最低限の自助手段が提供される。
- パスワード変更フローを用意することで定期更改が可能になる。
- email を対象外と明記することで本人確認フロー一式の実装を回避でき, スコープが抑えられる。
- 物理削除採用により実装が単純になり, 個人情報を残さないという観点でも素直。

代償・リスク:
- `User.postalCode` 列を追加するため Prisma マイグレーションが 1 本発生 (`add_user_postal_code`)。既存ユーザーは NULL で初期化される (任意項目のため OK)。
- 物理削除は誤操作時の復旧手段が無い。確認ダイアログ + 現パス再入力で予防するが完全ではない。
- email 変更を非対応とするため, 将来このルールを変える場合は別 ADR で再決定する。
- アカウント削除時に `Verification` の identifier=email レコードを別途削除する処理を忘れると, 古い確認トークンが残り得る。実装側で漏れないようテストでカバーする。

将来検討余地:
- Email 変更 (本人確認フロー込み) は別 US/ADR で扱う
- 2FA / Passkey 等の認証強化
- 削除ログ (監査用) の保管が必要になった際は論理削除 + 監査テーブルへ切り替え (本 ADR を Superseded する形)
