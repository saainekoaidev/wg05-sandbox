# 0009 駅マスタ取り込み時に駅番号と電報略号を区別する

## Status
Accepted (US-037 で採用)。ADR 0008 (駅番号を路線ごとに保持) を補完する。

## Context
ADR 0008 で `StationLine.code` に Wikidata P296 (station code) を取り込む方針を採用した。実装後の `--clean` 取り込みで実データを確認したところ、JR 路線の一部駅で **駅番号と電報略号 (telegraphic code) が同じ P296 statement に混在** していることが判明した。

例:
- 興津 (Q5359xxx, JR東海道線): `CA13/オキ/ヲキ`
- 蒲原: `CA11/カラ`
- 勝川 (Q5359249, JR中央本線): `CC01/カカ`

ここで `CA13` が現行の駅番号 (路線記号 + 数字), `オキ` `カカ` `ヲキ` が **電報略号** (旧国鉄時代の駅識別子。現在も保線・運転業務で使用)。

電報略号を駅番号として保持してしまうと:
- 駅マスタ管理画面 (S10) の駅番号入力欄に「カカ」が表示され, US-033 で導入した clientside 検証 (`^[A-Za-z0-9/-]*$`) を通らない。利用者は「保存しようとするとエラーになる駅」を編集できない。
- 駅マスタ参照 (S07) の「駅番号」列に電報略号が混じり, 文字列ソート (US-030) の結果も予測しづらい。

Wikidata 側で statement node の qualifier (`P31` = instance of, または `P518` = applies to part) を使って区別している例もあるが, 一貫しておらず信頼できない。一方で「電報略号は基本カタカナ 2-3 文字, 駅番号は半角英数 + 記号」という形式上の差は明確で安定している。

## Decision

import 時 (`fetchStationsForLines` 内) で各 `?stationCode` 値を採用するかどうかをアプリ層でフィルタする:

```ts
function isLikelyStationNumber(code: string): boolean {
  // 半角英数 + ハイフン + スラッシュ + ドット + アンダースコア のみで構成される値のみ採用
  return /^[A-Za-z0-9._/-]+$/.test(code)
}
```

これに合致しない値 (カタカナ/ひらがな/漢字/その他全角文字を含む値) は **電報略号または別種の識別子と判定して除外** する。qualifier 付きでも qualifier 無しでも同じフィルタを適用する。

副次的効果として:
- 既存の `Station.code` 検証 (US-033) と完全に整合する (画面上で編集できないコードは取り込みでも入らない)。
- 国際的な駅 (将来 JR 以外の海外路線を扱う際) にも同じルールで適用できる (アルファベット主体の駅番号体系であれば通る)。

### 既存データ
取り込み後の DB に既に電報略号が混入している場合は, ADR 0008 に従い `pnpm --filter backend exec tsx scripts/import-master-tokai.ts --clean` で再取込する (本 PR でも実行する)。

### テスト
- `import-master-tokai.test.ts` に電報略号フィルタのユニットテストを追加 (`カカ` / `オキ` 等が落ち, `CA68` `JY01` `CA66/CC00` が通る)。

## Consequences

### 利益
- 駅番号入力欄が常に編集可能 (バリデーション違反値が DB に入らない)。
- 駅マスタ参照のソート結果が安定 (英数値のみ)。
- ADR 0008 の駅番号 schema との整合性が取れる。

### 代償・リスク
- カタカナ/漢字を含む正規の駅番号体系を持つ事業者が将来現れた場合, 別途規約を見直す必要がある (現状の取り込みスコープ = 東海4県の鉄道 では問題なし)。
- 「カカ」のような電報略号も実務上は識別子として価値があるが、本サンドボックスでは扱わない (将来必要なら `StationLine.telegraphCode` 等を別カラムで持つ拡張が考えられる)。

### 影響範囲
- ADR 0008 §取り込み: 「qualifier 付き code → 該当 link.code, qualifier 無し → 全 link 共通」のロジックは維持。本 ADR はその前段の「採用する code 値の判定」を追加する。
