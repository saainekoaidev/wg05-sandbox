# 0003 デザインシステムの一次正 (CSS の重複解消方針)

## Status

Accepted

## Context

US-002 ログイン画面の実装 ([PR #8](https://github.com/saainekoaidev/wg05-sandbox/pull/8)) において、UI 設計フェーズ ([PR `e1c6000`](https://github.com/saainekoaidev/wg05-sandbox/commit/e1c6000)) で作成した HTML モックの共通スタイル [docs/ui-images/style.css](../ui-images/style.css) を、React 実装で利用するため [frontend/src/styles/app.css](../../frontend/src/styles/app.css) にそのまま移植した。結果として **同一内容の CSS が 2 ファイルに並存** している状態となり、PR #8 のセルフレビューで「M-1: CSS の二重管理」として指摘された ([PR #9 ではスコープ外として保留](https://github.com/saainekoaidev/wg05-sandbox/pull/9))。

このまま放置すると以下の問題が発生する:

- 設計と実装で同じ目的のスタイルを **2 ファイルで個別に編集** することになり、片側の変更が他方に伝播せず食い違うリスクがある
- US-003 以降の画面実装で同種の問題が増殖する可能性が高い
- CLAUDE.md の工程運用ルール「画面に関する判断は docs/ui.md に反映」とも整合性をとる必要がある

### 取り扱う対象

| ファイル | 役割 | サイズ |
| --- | --- | --- |
| `docs/ui-images/style.css` | 設計フェーズで作成した HTML モック (`docs/ui-images/*.html`) の共通スタイル | ~14KB |
| `frontend/src/styles/app.css` | React 実装が参照する共通スタイル (PR #8 で移植) | ~14KB (同一内容 + `.field-error` 追加) |

両ファイルが今後どう変化していくかが論点である。

### 制約

- HTML モック (`docs/ui-images/*.html`) は **ブラウザで単独で開ける** ことが価値の中心 (Figma を使わない代替として設計時の確認手段)
- 実 React アプリは Vite ビルドの一部として CSS を取り込む
- Windows / WSL / macOS 横断で動作する必要があるため、**シンボリックリンクは避けたい** (Windows での扱いが面倒)
- 教育用サンドボックスのため過剰な自動化は避け、運用ルールで吸収できるならそちらを優先

### 検討した代替案

1. **シンボリックリンク**: `docs/ui-images/style.css` を `frontend/src/styles/app.css` への symlink にする。一元管理できるが Windows 互換性問題および Git の symlink 管理の煩雑さで却下。
2. **ビルド時コピー**: ビルドスクリプトで一方を他方にコピー。CI ジョブが増え、2ファイル存在の根本問題が解決しない。却下。
3. **HTML モック側を実 CSS の相対パス参照に書き換え**: `<link rel="stylesheet" href="../../frontend/src/styles/app.css">` のようにする。技術的には可能だが、HTML モックを単独で開いたときに依存先が壊れやすく (移動すると path が崩れる)、また「mock は frontend に依存しない」という設計フェーズの独立性も失われる。却下。
4. **モックを廃止して app.css 一本化**: 実装完了後にモックを削除する案。設計時の参照価値を失うため却下。
5. **両者を並存させ、運用ルールで一次正と同期方針を明文化** (採用案): モックは設計時の凍結スナップショット、app.css が実装の一次正。
6. **CSS 変数や utility だけ別ファイルに切り出して両者で `import`**: モジュラ化は将来的に有効だが、現状の規模 (~14KB の単一ファイル) では過剰設計。将来の検討事項として残す。

## Decision

### 一次正 (Single Source of Truth)

- **`frontend/src/styles/app.css` を実装フェーズ以降の一次正とする**
- 実装中に新しいスタイル / 既存スタイルの修正が必要になった場合、変更は **`app.css` のみ** に対して行う
- HTML モック ([docs/ui-images/](../ui-images/)) は **設計フェーズの凍結スナップショット** として保持する。実装が始まったあとは原則更新しない

### モックを更新するケース

以下の場合のみ、`docs/ui-images/style.css` および対応する `*.html` を更新する:

1. **ユーザストーリーの新規追加** で大幅な画面追加が発生し、設計レビューのために新しいモックが必要なとき
2. **デザイン言語そのものの変更** (色 / フォント / shell パターン等) を行うとき (ADR 起票必須)

部分的な改善 (タイポ修正、軽微なバリデーション表示調整、コンポーネント追加程度) では更新不要。

### 食い違いが生じた場合

- `app.css` を真とする
- HTML モックを開いて表示が古いことに気付いた場合、それは仕様の進化であり、モックは「過去の設計時点の姿」を意味する
- 必要に応じて該当モックの先頭に「frozen at PR #N (YYYY-MM-DD)」コメントを書き加える

### 命名と運用

- `app.css` は `frontend/src/styles/` 配下に置く (現状維持)
- セクションコメント (`/* ボタン規則 */` 等) は両ファイルで対応関係が分かるよう揃える

### 採用根拠

| 観点 | 採用案の妥当性 |
| --- | --- |
| 一元管理 | 実装フェーズ以降は `app.css` 一本で扱う |
| 設計資産の保全 | モックは凍結スナップショットとして残るため、設計レビューの履歴が保たれる |
| 自動化コスト | ゼロ (運用ルールのみ) |
| Windows 互換性 | 影響なし (symlink を使わない) |
| 将来のリスク | モジュラ化が必要になった時点で別 ADR で再判断可能 |

## Consequences

### 利点

1. **CSS 修正時の単一参照点**: スタイル変更は `app.css` のみを対象にすればよく、漏れ・食い違いが構造的に発生しない
2. **設計履歴の保全**: HTML モックは凍結スナップショットとして、設計フェーズで合意した姿を再現可能な形で残せる
3. **シンプルさ**: 自動同期スクリプト等を導入しないため、教育用サンドボックスの「読みやすさ」を損なわない
4. **CLAUDE.md 工程運用ルールとの整合**: 画面の判断は [docs/ui.md](../ui.md) と個別 md (`docs/ui-screens/`) に集約する原則を維持しつつ、CSS 一次正だけ実装側に置く線引きが明確になる

### 代償

1. **モックと実装の見た目が時間とともに乖離する**: 実装が成熟するとモックは「設計時のあの姿」を残し、現物アプリと差が出る。これは設計履歴として許容する
2. **新規参加者がモックを最新と勘違いする可能性**: モックの先頭に「frozen at...」コメントを置く運用で軽減する (本 ADR 採択後に該当ファイルへ追記予定)
3. **モック単体で UX 確認しにくくなる**: 実装が進むほど、UI 確認には実 React アプリを起動する必要がある (`pnpm dev`)。これは MVP 規模の本サンドボックスでは大きなコストではない

### 将来のリスク

1. **画面数が増えると app.css が肥大化する**: 現状 ~14KB は許容範囲だが、US-003〜007 を実装すると 20〜30KB に成長する可能性。30KB を超えた段階で CSS Modules / Tailwind / 別ファイル分割を検討する別 ADR を起票する目安とする
2. **デザイン言語そのものの変更コストが高くなる**: 例えばカラーパレットを刷新する場合、`app.css` の `:root` 変数を変えるだけで全画面に伝播するが、モックも追従させるか凍結のままにするかは都度判断
3. **将来の design system パッケージ化**: `frontend/src/styles/` を独立パッケージ (例: `@wg05/ui`) として切り出したい場合は本 ADR の後継 ADR で扱う

### 他の決定への波及

1. **PR review 観点**: 今後 `docs/ui-images/*.css` が変更されている PR を見たら「ADR 0003 のどのケースに該当するか」を確認する
2. **新規画面実装時のフロー**: US-003 経路登録などで新しいスタイル要素 (例: `.segment-card-head`) が必要になったら `app.css` に追加し、HTML モックは更新しない
3. **ADR 0002 (データモデル) との関係**: 直接の依存関係はなし

### 対応アクション (本 ADR と並行で実施)

- [ ] [docs/ui-images/style.css](../ui-images/style.css) の先頭に "frozen at PR #8 / ADR 0003" コメントを追加 (将来の混乱防止)
- [ ] [docs/ui.md](../ui.md) の「§4.2 画面構成」または該当箇所に、本 ADR への参照を追記
- [ ] CLAUDE.md「工程運用ルール」に「CSS は frontend/src/styles/app.css を一次正とする (詳細は ADR 0003)」を追記

## 未確定事項

- HTML モックを「frozen at」コメントで凍結マークする際の文言の最終形 (本 ADR では運用ルールとして決め、具体ファイル更新は別タスク)
- モック更新が許容される境界の具体線 (現状「ユーザストーリー新規追加時」「デザイン言語の変更時」と粗く定義しており、実運用で揺らぐ可能性あり)
- 将来の design system 独立化のタイミング判断基準
- CSS テストの戦略 (visual regression の要否)
