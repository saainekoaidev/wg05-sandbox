# 0012 同一物理駅の Wikidata エンティティ分割を取り込み時にマージする

## Status
Accepted (US-041 で採用)。ADR 0008 / 0010 / 0011 (駅番号取り込みシリーズ) を補完する。

## Context
Wikidata では同じ物理駅が複数の Q-ID に分割されているケースが頻発する。特に「JR/名鉄駅 と 地下鉄駅 が同名同位置だが別エンティティ」というパターン。

実例:

- **大曽根駅**:
  - Q872075 = JR中央本線 + 名鉄瀬戸線, P131=名古屋市, 座標(136.937, 35.19)
  - Q110831202 = 名城線, P131=名古屋市, 座標(136.937, 35.19), **P138 = Q872075** (名前の由来)
- **名古屋駅** (5 エンティティに分割):
  - Q1109915 = 近鉄名古屋線, P131=名駅一丁目
  - Q911757 = 名鉄名古屋本線, P131=名駅一丁目
  - Q110799384 = JR中央/関西本線, P131=名駅一丁目
  - Q56611989 = 桜通線/東山線 (地下鉄), P131=名古屋市
  - Q124417968 = あおなみ線, P131=名駅一丁目
  - 全 5 エンティティが半径 約 300m 以内 (座標 136.881-136.884, 35.169-35.171)

現行 import は Q-ID 単位で 1 Station 行を作るため、駅マスタ管理 (S10) や 駅マスタ参照 (S07) で同名駅が複数行に並んでしまい:
- 利用者が経路登録 (S04) で「大曽根」を選ぶ時にどれを選ぶか迷う
- 駅番号 (US-040 の prefix routing) も Q-ID 内に閉じて学習されるため、複数 Q-ID 間で情報が伝播しない

## Decision

### マージ条件
取り込み中の `acc` 集約後、以下のいずれかが成立する 2 つの Station エンティティを **同一物理駅** とみなしてマージする:

- **(a) P138 リンク**: 一方が他方を P138 (名前の由来 / named after) で参照している。Wikidata 編集者が明示した強いシグナル。
- **(b) 同名 + 座標近接**: 正規化済み name (ADR 0007 §normalizeStationName 適用後) が一致 + 座標距離 < **500m** (Haversine 概算)。座標は P625 で取得。座標が片方でも欠落していたらこの条件は不成立。

両方成立する場合は (a) を優先 (canonical 選定で P138 ターゲットを使う)。

実装上は Union-Find で連結成分を計算し、同一成分内の全エンティティを 1 駅にマージする。

### canonical Q-ID 選定
グループ内で 1 つの Q-ID を canonical として選び、Station レコードはその ID で 1 つだけ作成する。優先順:

1. 他のグループメンバーから P138 で参照されている Q-ID (= 元エンティティ)
2. 1 が無い場合: lineLink 数が最大の Q-ID
3. 同点の場合: Q-ID の lex 昇順で先頭

### マージ後のフィールド
- `name`: canonical の name を採用 (既に normalize 済み)
- `kana`: canonical → 他エンティティの順で最初の非空文字を採用
- `sourceUri`: canonical の Wikidata URL のみ保持 (他は捨てるが、merge ログで Q-ID は記録)
- `lineIds` (P81 の集約結果): 全エンティティの union
- `codesByLine` (qualifier 付き P296): 全エンティティの per-line union
- `unattachedCodes` (qualifier 無し P296): 全エンティティの union

prefix 学習 (US-040 / ADR 0011) は **マージ後** の集約データに対して走るため、merge により学習データが充実する副次効果もある。

### SPARQL クエリの拡張
`fetchStationsForLines` の SELECT に追加:
- `?coord` (P625): 座標
- `?namedAfter` (P138): P138 リンク

### 既存データ
ADR 0008 系と同じく `--clean` 再取込で補正。

### テスト
`import-master-tokai.test.ts` に以下を追加:
- 大曽根パターン: P138 リンクで 2 Q-ID がマージされる
- 名古屋パターン: 同名 + 近接座標で複数 Q-ID がマージされる
- 同名だが遠隔 (座標距離 > 500m) の駅は別 Station として残る
- 一方の座標が欠落している場合は (b) 条件不成立 → 別駅扱い

## Consequences

### 利益
- 駅マスタ管理 (S10) で 大曽根 / 名古屋 等の同名物理駅が 1 行に統合される。
- 駅マスタ参照 (S07) の検索結果が 1 件になり、利用者の選択迷いが解消。
- 路線情報・駅番号情報がマージ後の単一エンティティに集約され、prefix 学習 (US-040) のカバレッジ向上にも寄与。
- Station 件数が 804 → 概算 700 強に削減 (DB 負荷低減)。

### 代償・リスク
- 同名 + 座標 < 500m の異なる駅 (実在性は低いが理論上ありうる) を誤マージする可能性。本サンドボックスの東海4県スコープでは確認できなかった。
- 座標が欠落している駅は (b) 条件で merge されない。Wikidata 側で座標を埋めれば自動的に解消する。
- 既存 `Station.id` (Q-ID) を使った外部連携 (現状なし) があれば破壊的変更。本サンドボックスでは Station.id は Wikidata 由来 ID を内部的に使うのみで API レスポンスでも露出しているが、マスタ取り込みは `--clean` 前提なので互換性維持の必要なし。

### 影響範囲
- `fetchStationsForLines` に merge step を追加 (acc 集約後 + prefix 学習前)。
- prefix 学習・disposition ロジック (ADR 0011 / 0010) はそのまま、merge 済データに対して動作する。
- 駅マスタ参照 (S07) / 経路登録 (S04) では駅選択肢の数が減る (利用者にとっては改善)。
