# 0006 マスタ管理機能の権限モデルと参照整合性

## Status
Accepted (2026-05-03)

## Context

US-012 (路線マスタ CRUD) と US-013 (駅マスタおよび駅↔路線接続 CRUD) を画面から行えるようにすることが要件として追加された。これにより以下を決める必要がある:

1. **権限モデル**: マスタは全ユーザーの経路登録に影響するため, 一般ユーザーが自由に編集できる状態は危険。「管理者」を導入するか, それとも CLI/seed 経由のみに留めるか。
2. **参照整合性**: 既存の `RouteSegment.lineId` は `onDelete: SetNull` ([backend/prisma/schema.prisma:89](backend/prisma/schema.prisma#L89)) なので路線削除自体は可能。だが UX として「いきなり既存経路の路線名が消える」のは望ましくない。削除時の挙動 (拒否/警告/SetNull のどれ) を決める必要がある。
3. **駅名の整合性**: 駅マスタの name を変更しても既存の `RouteSegment.fromStation/toStation` (文字列複製) は更新されない設計 (ADR 0002 §3.2.2)。これは意図的な設計なのか, 運用上の不整合とみなして追従させるのか。

ADR 0005 で取り込みソースを Wikidata に決めたが, Wikidata の精度は 100% ではないため運用補正の窓口として画面 CRUD を提供する必要性自体は確定している。

## Decision

### 1. 権限モデル

`User` モデルに **`role String @default("user")`** 列を追加する。値は `"user"` / `"admin"` の 2 値。

- 新規サインアップ (US-001) は常に `role: "user"`
- 管理者の昇格は **CLI のみ** (`backend/scripts/grant-admin.ts <email>`)。画面からの昇格 UI は本 ADR では設けない (権限管理画面を作ると要件が膨らむため)。
- マスタ管理画面 (`/admin/lines`, `/admin/stations`) は role=admin のみアクセス可。一般ユーザーがアクセスすると 403 + 「管理者権限が必要です」を表示。
- API 側も同様 (`POST/PUT/DELETE /api/lines`, 同 `/api/stations` は role 検証ミドルウェア必須)。

代替案として「管理者は廃止して CLI のみ」も検討したが, ADR 0005 で取り込んだ Wikidata データの誤りを発見した利用者が即座に補正できる窓口は必要なため, 画面 CRUD を採用した。

### 2. 路線削除時の挙動

**「該当路線を参照している RouteSegment が存在する場合は削除を拒否」** とする。すなわち UI 側で削除ボタン押下 → API 側で参照件数チェック → 0件なら delete, 1件以上なら 409 Conflict を返し, レスポンスに参照件数とサンプル経路 ID リスト (最大 5 件) を含める。

`onDelete: SetNull` の挙動は schema 上は維持する (DB 強制制約は必要に応じて緩く保つ) が, アプリ層で事前検証することで「気づかぬうちに既存経路が壊れる」事態を防ぐ。

代替案:
- **SetNull (現行)**: 削除自体は通るが UX 的に「路線が突然消える」現象が起きる。利用者が原因を追えない。
- **強制 Restrict (DB 制約)**: スキーマレベルで削除不可。だが管理者が「もう廃線になった路線を整理したい」場合の UX が悪い (既存経路を全部修正してから再削除, 等)。
- **採用案 (アプリ層 Conflict)**: 409 + 詳細レスポンスにより, 管理者が「この経路を直せば消せる」と判断できる。

### 3. 路線編集時の挙動

`Line.name` / `Line.kind` / `Line.operator` は自由に編集可能。既存 RouteSegment は外部キー (lineId) で繋がっているため, 編集後も画面側は自動的に新しい name で表示される (LINE_BY_ID lookup が反映される)。

`Line.id` の変更は不可 (主キー変更は副作用が大きすぎるため)。誤った id で取り込まれた場合は「削除 → 新規作成」で対応。

### 4. 駅マスタ削除時の挙動

**RouteSegment 側の `fromStation` / `toStation` は文字列複製であり外部キーではない** ため, 駅マスタを削除しても既存経路は壊れない。よって駅は **無条件で削除可能** とする (路線と挙動が異なる)。

ただし「StationLine (駅↔路線接続)」のレコードは Cascade で連動削除する。

### 5. 駅マスタ編集時の挙動

`Station.name` を変更しても既存 RouteSegment の `fromStation` / `toStation` 文字列は **追従させない**。これは ADR 0002 の「駅名はフリー入力許容のため文字列で複製保存する」設計を維持するため。利用者が古い駅名を経路として残しているケースを尊重する (駅名改称があった場合でも履歴的に残ることに価値がある)。

その代わり, 駅マスタ編集画面に注意書き「既存経路に登録されている駅名文字列は自動更新されません」を表示する。

### 6. 駅↔路線接続の管理

`StationLine` の追加・削除は駅マスタ編集画面内のサブセクションで行う (専用画面は設けない)。1 駅に対し複数の路線をチェックボックスで紐付ける UI とする。

### 7. 監査ログ

本 ADR では **監査ログを設けない**。サンドボックスの工程追体験という目的に照らし不要と判断。将来要件として追加された場合は別 ADR (or 本 ADR の Superseded) で扱う。

### 8. 同時編集の扱い

US-006 (経路編集) では `Route.updatedAt` による楽観ロックを採用したが, マスタ管理は管理者のみが触る前提のため楽観ロックは導入しない。代わりに「最後の更新が勝つ (Last Write Wins)」とする。実運用で問題化したら別 ADR で見直す。

## Consequences

利点:
- 一般ユーザーがマスタを破壊できないことが保証される (role による分離)。
- 路線削除時の参照チェックにより, 「気づかぬうちに既存経路が壊れる」事態が防げる。
- 駅マスタは無制約に編集できるため, Wikidata 取り込みの誤りを管理者が画面ですぐ補正できる。
- 監査ログを設けないことで実装スコープを抑えられる。

代償・リスク:
- `User.role` 列追加によるマイグレーションが 1 本発生。既存ユーザーは default("user") で初期化されるため移行コストは低い。
- 管理者昇格を CLI のみとしたため, 初回のセットアップ手順 (シード後に `pnpm exec tsx scripts/grant-admin.ts admin@example.com`) を README/docs に明記する必要がある。
- 駅名改称時に既存経路の文字列が追従しないため, 利用者から見て「マスタを直したのに表示が古い」と誤解される可能性。注意書きで予防する。
- 路線削除拒否のフローで管理者が「先に経路を全部削除してから路線を消す」必要が出るケースが出てくる。UX として煩雑だがデータ整合性を優先する。
- 監査ログ無しのため, 「誰がいつマスタを変えたか」が追跡できない。サンドボックスでは許容範囲だが商用利用時は不可。
- Last Write Wins のため, 複数管理者が同時に編集すると後勝ちで上書きされる。管理者数が少ない前提で許容する。

将来検討余地:
- 監査ログ (操作履歴テーブル)
- 管理者権限の画面付与 UI
- マスタ更新時の楽観ロック
- 駅名追従モード (既存経路の文字列を一括置換するメニュー)
