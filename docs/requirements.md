# 要件定義

US-001	ユーザとして, メールアドレスとパスワードでアカウントを新規登録したい。なぜなら自分専用に通勤経路を保存したいから。
US-002	ユーザとして, メールアドレスとパスワードでログインしたい。なぜなら登録済みの経路を確認･編集したいから。
US-003	ユーザとして, 出発駅と到着駅と経由駅と区間運賃を入力して通勤経路を登録したい。なぜなら毎月の交通費を可視化したいから。
US-004	ユーザとして, 自分が登録した通勤経路を一覧で見たい。なぜなら登録済みの経路をひと目で把握したいから。
US-005	ユーザとして, 1経路の詳細情報を見たい。なぜなら経由や区間ごとの内訳を確認したいから。
US-006	ユーザとして, 既存の通勤経路を編集したい。なぜなら住所変更等で経路が変わった場合に更新したいから。
US-007	ユーザとして, 不要になった通勤経路を削除したい。なぜなら過去の不要データを整理したいから。
US-008	ユーザとして, 自分の登録済みプロフィール (氏名/郵便番号) を変更したい。なぜなら引っ越しや改姓などのライフイベントで情報が変わるから。
US-009	ユーザとして, 自分のパスワードを変更したい。なぜなら定期的なパスワード更改や情報漏洩懸念時に自衛したいから。
US-010	ユーザとして, 自分のアカウントを削除したい。なぜならサービスを利用しなくなった際に登録情報を残したくないから。
US-011	システムとして, 東海4県 (愛知/岐阜/三重/静岡) の鉄道路線および駅マスタを取り込んでおきたい。なぜなら本サンドボックスが想定する利用エリアの経路を網羅的に登録できるようにするため。
US-012	管理者として, 路線マスタを画面から登録/修正/削除したい。なぜなら US-011 の取り込み後に実態と乖離した路線情報を運用で補正したいから。
US-013	管理者として, 駅マスタおよび駅↔路線の接続情報を画面から登録/修正/削除したい。なぜなら新駅開業/廃駅/接続変更を反映したいから。
US-014	ユーザとして, ヘッダ/フッタなど自分の識別を示す箇所には登録した氏名 (User.name) を表示してほしい。なぜならメールアドレスは ID 的な意味合いが強く、ログイン中アカウントを直感的に確認したい場面では氏名のほうが自然なため。
US-015	ユーザとして, プロフィール設定画面 (S08) のパスワード変更欄および削除確認用パスワード欄に, 画面表示時にブラウザの自動補完で値が埋まっていないでほしい。なぜなら入力していないパスワードが表示されているのは違和感があり, 誤操作 (そのまま「変更する」「削除する」を押下) の温床にもなるため。
US-016	ユーザとして, 経路登録/編集 (S04 / S06) の各区間で「種別」「路線」を選択した状態で「駅選択」ボタンを押下した時, 駅マスタ参照 (S07) が選択済み条件で絞り込んだ検索結果を最初から表示している状態で開いてほしい。なぜなら毎回同じ条件を popup 側で入力し直すのが煩わしいため。
US-017	ユーザとして, 駅マスタ参照 (S07) の検索フォームで種別を選択した時に路線リストの選択肢がその種別の路線のみに絞られ, 逆に路線を選択した時に種別もその路線が属する種別に自動で揃うようにしてほしい。なぜなら矛盾した条件 (例: 種別=バスで地下鉄路線) で空振り検索することを防ぐため。
US-018	システムとして, JR が運営する路線の表示名は「JR○○線」「JR○○本線」のように "JR" プレフィックスを付けて表示したい。なぜなら Wikidata 由来の素の名称 (中央本線, 関西本線, 等) では他事業者線と区別しづらいため。
US-019	ユーザとして, ログイン中の自分の氏名 (User.name) をログイン/サインアップ画面以外のすべての画面でヘッダ右上に常時表示してほしい。なぜなら現在ログインしているアカウントを画面に関わらず即座に確認したいから。「ユーザー：」のラベルは不要で氏名のみ表示し、クリックでアカウント設定 (/account) に遷移することで US-014 の動線を維持する。
US-020	ユーザとして, 経路登録 (S04) / 経路編集 (S06) の各区間でも、種別を選択すると路線セレクトの選択肢が該当種別のものだけに絞られ, 路線を選択すると種別がその路線の種別に揃う cascade 連動が効いてほしい。なぜなら US-017 で駅マスタ参照画面に入れた仕組みと同じ整合性を経路入力でも維持したいから。
US-021	ユーザとして, 経路一覧 (S03) のヘッダで右上の UserBadge と「+ 新規登録」ボタンが視覚的に重ならないようにレイアウトを調整してほしい。なぜなら現状ボタンとバッジが重なって読みにくいから。
US-022	ユーザとして, 経路詳細 (S05) のカード幅を広げ、「一覧に戻る」ボタンのラベルが改行されないようにしてほしい。「編集」「削除」ボタンの幅も詰まりすぎないように適度に広げ、画面が縦長すぎず読みやすくなるようにしたい。
US-023	システムとして, 駅マスタの kana 列に漢字駅名 (フォールバック値) が混入しないようにしてほしい。Wikidata に kana 情報が無い駅は kana を空にする。なぜなら検索画面で kana が漢字表記で出るのは利用者に違和感を与えるから (本格的な漢字→ひらがな変換は別途検討)。
US-024	ユーザとして, プロフィール設定 (S08) の ADMIN セクションの説明文 (「管理者として以下のマスタ管理画面にアクセスできます」) を他のセクションの hint と同じ位置 (左インデント) に揃え, マスタ管理ボタン 2 つを少しコンパクトにしてほしい。なぜなら現状はテキストの位置が左端に寄りすぎて違和感があり, ボタンも横幅が広すぎてセクションのバランスが崩れているから。
US-025	管理者として, 路線マスタ管理 (S09) の新規作成と編集を専用画面 (URL) で行いたい。なぜなら現状のインライン展開フォームよりも独立画面のほうが入力に集中しやすく, 戻る操作も明示的に行えるから (新規 = `/admin/lines/new`, 編集 = `/admin/lines/:id/edit`)。
US-026	管理者として, 駅マスタ管理 (S10) の新規作成と編集を専用画面 (URL) で行いたい。理由は US-025 と同じ (新規 = `/admin/stations/new`, 編集 = `/admin/stations/:id/edit`)。
US-027	システムとして, US-011 で取り込んだ駅マスタの大部分で kana が空になっているところに自動でひらがなを入れたい。なぜなら駅マスタ参照のかな検索 (US-003 想定) が空欄駅では効かず利用体験が低下するから。Wikidata に kana が設定されている駅はそれを使い、無い駅は駅名漢字を形態素解析等でひらがな化して埋める。末尾の「えき」は付けない。
US-028	ユーザとして, 駅マスタ管理 (S10) および路線マスタ管理 (S09) の操作列で「編集」と「削除」ボタンが行高内で横並びになるようにしてほしい。なぜなら現状はセル幅が狭いと縦に折り返されて見栄えが悪いから (`.col-actions` の flex-wrap を nowrap に変更)。
US-029	ユーザとして, 路線マスタ管理 (S09) / 駅マスタ管理 (S10) のフッタから「経路一覧へ」リンクは取り除いてよい。なぜならアカウント設定経由でも経路一覧に戻れるため重複動線である。フッタは「アカウント設定」のみ残す。
US-030	システムとして, 駅マスタに駅番号 (Wikidata の P296 = station code) を取り込み, 駅マスタ参照画面 (S07) で「種別」「よみがな」「駅番号」のいずれかで昇順/降順ソートできるようにしたい。なぜなら実際の駅の並び順 (路線上の駅番号順) で確認したい場面があるから。
US-031	管理者として, 路線マスタ管理 (S09) の一覧を種別 (電車/地下鉄/バス/その他) で絞り込みたい。なぜなら登録路線数が増えるとリストが長くなり, 見たい種別だけ確認しづらいから。
US-032	管理者として, 駅マスタ管理 (S10) の一覧を種別 + 路線で絞り込みたい。なぜなら US-011 取り込み後は 800 駅超で一覧が膨大になり, 編集対象の駅にたどり着くのに時間がかかるから。種別は cascade で路線セレクトを絞り (US-017 と同じ挙動), 路線も種別もすべて で全件表示。
US-033	管理者として, 駅マスタ管理 (S10) の新規/編集画面で各駅に「路線ごとの駅番号」を表示・入力したい。なぜなら駅番号は実体として路線ごとに採番される (例: 名古屋駅は東海道線で CA68, 中央線で CC00) ものであり, 駅単位の単一フィールドでは正しく表現できないから。データ構造として `Station.code` は廃止し, `StationLine.code` を新設する。UI では接続路線セレクト (チェックボックス) の各行に駅番号入力枠を同居させ, チェックを外すと駅番号入力もクリアされる。チェック列の幅はチェックボックス本体に合わせて狭くする。
US-034	管理者として, 路線マスタ管理 (S09) と 駅マスタ管理 (S10) の絞り込みフィルタ (種別 / 路線) を新規・編集画面に遷移して戻ってきたときに維持しておいてほしい。なぜなら US-031/032 で追加したフィルタは画面遷移ごとに初期化されてしまい, 同じ駅/路線群を続けて編集する作業フローで毎回フィルタを掛け直す必要があるから。戻ってきたときにはその維持されたフィルタ条件で一覧を再取得して最新状態を表示する。永続範囲はセッション (sessionStorage) で十分。
US-035	管理者として, 駅の新規登録・編集画面の「接続路線 / 駅番号」リストで, 路線名が縦に折り返されず 1 行で表示されるようにしてほしい。なぜなら現状は駅番号入力枠が広すぎて路線名側の幅が狭まり「上飯田線」が「上 / 飯 / 田 / 線」のように 1 文字ずつ縦並びになり読みづらいから。駅番号入力枠を狭めれば路線名 1 行表示にできる。
US-036	管理者として, 駅マスタ管理 (S10) の一覧表で駅名・よみがなを 1 行で表示してほしい。行の高さが広がる要因は接続路線/駅番号の数が 1 行に収まらない場合だけにしたい。なぜなら現状は接続路線セルの幅が広く確保されすぎて, 同じ行の駅名・よみがなが影響を受け縦長になることがあるから。
US-037	システムとして, 駅マスタ取り込み時に「駅番号」(例: CA68) と「電報略号」(例: カカ, オキ, ヲキ) を区別し, 電報略号は StationLine.code に含めないでほしい。なぜなら Wikidata の P296 (station code) には JR 駅では駅番号と電報略号 (カタカナの古い識別子) が混在しており, 取り込み後の駅番号入力欄に電報略号が紛れ込むと, 画面上で駅番号バリデーション (US-033 の半角英数+ハイフン/スラッシュ) を通らず編集できないから。判定は「半角英数 + ハイフン/スラッシュ/ドット/アンダースコアのみで構成される値のみ採用」とし, それ以外 (カタカナ/ひらがな/漢字を含む値) は除外する。
US-038	管理者として, 駅マスタ管理の新規/編集 (および路線マスタ管理の新規/編集) でフォーム送信時にバリデーションエラーが発生したら, エラーの原因となった入力枠やドロップダウンを視覚的に強調表示し, 自動でその位置にスクロール + フォーカスしてほしい。なぜなら現状はエラーメッセージがフォーム上部のバナーにのみ出るため, 該当の入力欄が画面外にあるときに「どこを直せば良いか」を探す必要があるから。実装は対象 input/select に `aria-invalid="true"` + `.is-error` クラスを付与し, CSS で red border + ring を出す。
US-039	システムとして, 駅マスタ取り込み時に複数路線が乗り入れる駅で「駅番号が路線をまたいで併記」されないようにしてほしい。なぜなら駅番号は実体として 1 駅×1 路線 に対して 1 つだけ採番されるものであり (例: 千種 = JR中央本線で CF03, 地下鉄東山線で H12), 現行の import は qualifier 無しの P296 値を「全 lineLink に同じ値を流し込む」fallback で処理するため複数路線駅で `CF03/H12` のように両路線に同じ併記が入ってしまうから。修正方針: SPARQL クエリに qualifier `P518` (applies to part) を加えて Wikidata の路線対応 qualifier を漏れなく取得 + 「未埋め lineLink が 1 件 + unattached code 1 件」のケースで残り 1 lineLink に自動割当する補完ロジックを入れる。残ったあいまいケース (未埋め lineLink ≥ 2 + unattached ≥ 1) は自動採用しない。複数路線駅の手動補完は原則不要。
US-040	システムとして, US-039 の補完では救えなかった「両 unattached + 両 unfilled」の駅 (例: 大曽根 = `CF04` + `ST06`) でも, 駅番号の prefix (英字部分) から路線を推定して自動割当してほしい。なぜなら `CF04` の prefix `CF` は中央本線記号, `ST06` の `ST` は名鉄瀬戸線の記号で、人間には自明に区別がつくのに自動取り込みで空のままになるのは捜索が甘いから。実装方針: 取り込み中の SPARQL 結果から qualifier 付きで紐付いた code を学習データとして「路線 → prefix 集合」を動的構築し, unattached code はその prefix で該当 lineLink に割当する。
